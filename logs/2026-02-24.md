### Proposal: DH — A Data Hub Command-Line Tool

Rufus proposed two CLI tools as the default interface for AI agents and humans alike. The first is **`DH`** (Data Hub) — for pushing data directly to Data Hub. The second is a separate **Data Presser** tool for data preparation and wrangling (which is rapidly becoming AI/skill-based work). The key architectural insight: bypass GitHub sync entirely. Instead of pushing to GitHub which then syncs to Data Hub, just push files straight to R2 storage. This is exactly what the Flower Show CLI already does — push a file, it creates a project, it goes live.

**MVP vision:** `dh push mydata.csv` → creates a project, stores the file on R2, and instantly serves a nice CSV viewer. Billing would be a passthrough on R2 costs with a small markup. Versioning is critical — Rufus wants something git-like but lighter (no full git protocol, which chokes on large data files). Ideally content-addressable storage where files are hashed and chunked, giving you commit-like transactionality: a push either completes or fails, and you get a change log you can walk through. He referenced Hugging Face's acquisition of Xet (2024) for their content-defined chunking approach, but suggested starting simpler — just store files with head pointers, iterate from there.

Rufus and Anuar agreed to do a deeper spec session where they dictate to AI simultaneously to flesh out the DH tool design. In the meantime, the priority is to identify pain points in the current Data Hub publishing flow — publishing a blog post on Data Hub right now is "really painful."

### Completing the Build-Ship-Learn Cycle

Rufus raised a recurring pattern: features get built but never fully complete the lifecycle of deploy → announce → learn → iterate. Two examples from Flower Show: the drag-and-drop file upload was built inside an A/B test, then the A/B test was removed, and the feature was never actually deployed or announced. Similarly, the Obsidian Dataview-compatible "basis" syntax support was implemented but the docs were so weak that Rufus — the first person to actually try using it — couldn't figure out the required markdown setup.

The conclusion: for bigger features, the team should ship within a week, define clear hypotheses, announce properly (not just a few Reddit posts — a framed story with a good video), establish metrics, and then decide: are we developing this further or throwing it out? Queryless is actually doing this well right now — prototype shipped, traffic measured, consolidation plan proposed.
